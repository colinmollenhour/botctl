# Getting started

`botctl` is a Rust CLI for launching, observing, and safely driving Claude Code sessions inside `tmux`.

For end-to-end operator flows, see [workflows](./workflows.mdx). For command details, see [command reference](./command-reference.mdx).

## Learn these first

Most users only need three commands:

- `dashboard` for a live control surface across Claude panes
- `yolo` for automated babysitting and recovery
- `serve` for a long-lived observation stream

Everything else is support around setup, diagnostics, prompt handoff, and lower-level recovery.

If you just want to get value quickly, install it, open `dashboard`, then try `yolo` or `serve`.

## Requirements

- `tmux`
- `claude` on `PATH`
- `botctl` installed

## Install

```bash
cargo install botctl
```

## Fastest useful path

Open the dashboard:

```bash
botctl dashboard
```

Start YOLO on one pane:

```bash
botctl yolo --pane 0:6.0
```

Stream live session events:

```bash
botctl serve --session demo --format jsonl
```

## First session

Start a managed Claude session:

```bash
botctl start --session demo --cwd /path/to/project
```

Then check what `botctl` sees:

```bash
botctl list
botctl doctor --session demo
botctl status --pane %19
```

Pane-targeted commands also accept tmux's explicit pane syntax:

```bash
botctl status --pane 0:2.3
```

Attach with tmux when you want the full terminal UI:

```bash
tmux attach -t demo
```

## If Claude is already running in tmux

If Claude was already started inside tmux, resolve and verify the pane explicitly before acting:

```bash
botctl list --all
botctl attach --pane %19
```

If you already know the tmux pane target, you can use it directly:

```bash
botctl attach --pane 0:2.3
```

## Dashboard and YOLO

Open the cross-workspace dashboard TUI:

```bash
botctl dashboard
```

Keep it alive in a dedicated tmux-backed popup session:

```bash
botctl dashboard --persistent
```

If you use tmux popups, this is the quick and easy way to make the dashboard feel built-in:

```tmux
bind-key C-c display-popup -E -w 80% -h 40% botctl dashboard --persistent
```

Enable YOLO for one explicit pane:

```bash
botctl yolo --pane 0:6.0
```

Enable YOLO for all panes in one workspace:

```bash
botctl yolo --all --workspace .
```

## Observe versus serve

- `observe` is bounded and returns a fixed snapshot/event sample.
- `serve` is the current foreground observer for one tmux session.
- `serve` is not a full daemon/API/SSE product yet.

If you are not sure which one you want, you probably want `serve`, not `observe`.

Run `serve` against one tmux session:

```bash
botctl serve --session demo
```

Watch one specific pane only:

```bash
botctl serve --session demo --pane %19
```

Machine-readable stream:

```bash
botctl serve --session demo --format jsonl
```

## Safety rules

- Prefer explicit pane IDs for automation.
- `session:window.pane` is also accepted when it identifies one pane exactly.
- Never automate an ambiguous target.
- Claude ownership is validated before automation runs.
- `Unknown` is a refusal state, not a guess.
- Folder trust approval is special and uses raw `Enter`.

For blocker recovery, prompt handoff, and recovery examples, see [workflows](./workflows.mdx) and [prompt handoff](./prompt-handoff.mdx).

## Current limits

- live classification is still built around `capture-pane`, with `serve` using a best-effort merged stream model when that helps break `Unknown` states
- the classifier is conservative and keyword-based
- `serve` is currently a foreground observer, not the full daemon/API system yet
